# v0.0 Prerequisites: Foundation for the Retrieval Substrate

Everything that must be in place before v0.1 can start. Infrastructure, refactoring, and migration work that the substrate depends on.

---

## 1. Scope

| Work | Why it's a prerequisite |
|------|------------------------|
| `_system_` space on every branch | Recipes, expansion cache, statistics need branch-scoped system storage |
| Default recipe stored in `_system_` space | The substrate reads the recipe from the branch. Must exist before `retrieve()` can run. |
| Recipe types and deserialization | The substrate needs to parse recipe JSON before it can execute anything |
| Move search config from `strata.toml` to recipe | BM25 k1/b, auto_embed settings currently in TOML. Must migrate so the recipe is the single source of truth. |
| Refactor existing search to use Searchable trait uniformly | Current hybrid search orchestrator bypasses primitives. The substrate fans out to primitives — they must work. |
| Move shadow collections from `_system_` branch to `_system_` space | Embeddings must be branch-scoped for the recipe's `vector.collections` to resolve on the current branch. |
| `set_recipe()` / `get_recipe()` API | Users need to set and retrieve recipes before `db.search()` can use them. |

## 2. `_system_` Space on Every Branch

**Issue:** #2119

### What it is

A hidden space named `_system_` on every branch. Uses existing space infrastructure. Created automatically when a branch is created.

### Properties

- **Hidden from user data operations.** `db.kv.list()`, `db.kv.scan()`, `db.json.list()` do not return `_system_` space data. The space is filtered from all user-facing list/scan/count operations.
- **Writable only by Strata internals.** `set_recipe()`, auto-embed hooks, statistics collection write here. Direct user writes to `_system_` space are rejected.
- **Forks via COW.** When a branch is forked, the `_system_` space is inherited from the parent just like any other space. Override on the fork, parent unaffected.
- **Uses existing space mechanics.** No new storage primitives. The `_system_` space is just a space with a reserved name and access restrictions.

### Implementation

**File:** `crates/engine/src/primitives/` (each primitive's space validation)

**Changes:**
1. **Space validation:** Add `_system_` to the reserved space names (alongside `_graph_` which is already reserved). Reject user writes to `_system_` via normal put/set APIs.
2. **Space filtering:** In `list()`, `scan()`, `keys()` for each primitive — skip entries in the `_system_` space unless called with an internal flag.
3. **Branch creation:** Ensure `_system_` space is available on new branches (no explicit creation needed — spaces are implicit in Strata's key prefix model).
4. **Internal write API:** Add `put_system()` / `set_system()` methods (or an internal flag) that bypasses the space restriction for Strata's own writes.

**Estimated effort:** ~200 lines across primitives. Mostly validation and filtering.

## 3. Default Recipe

### What it is

A recipe stored in the `_system_` space on every branch at a well-known key. The substrate reads this when `db.search()` is called without an explicit recipe.

### Storage

- **Key:** `_system_` space, KV key `recipe:default`
- **Value:** Recipe JSON (serialized via serde_json)
- **Named recipes:** `recipe:{name}` for user-defined named recipes
- **Created automatically:** When a database is opened and no default recipe exists, write the built-in defaults.

### Built-in default recipe

```json
{
    "version": 1,
    "retrieve": {
        "bm25": {
            "k": 50,
            "k1": 0.9,
            "b": 0.4,
            "field_weights": {"body": 1.0},
            "stemmer": "porter",
            "stopwords": "lucene33",
            "phrase_boost": 0.0,
            "proximity_boost": 0.0
        }
    },
    "fusion": {
        "method": "rrf",
        "k": 60
    },
    "transform": {
        "limit": 10
    },
    "control": {
        "budget_ms": 5000
    }
}
```

BM25 only. No vector, no graph, no expansion, no reranking. Works without models, without embeddings, without any configuration. The floor.

### API

```rust
// Engine level
impl Database {
    pub fn set_recipe(&self, branch_id: &BranchId, name: &str, recipe: &Recipe) -> StrataResult<()>;
    pub fn get_recipe(&self, branch_id: &BranchId, name: &str) -> StrataResult<Option<Recipe>>;
    pub fn get_default_recipe(&self, branch_id: &BranchId) -> StrataResult<Recipe>;
}
```

`get_default_recipe()` returns the recipe at `recipe:default` in the `_system_` space. If none exists (fresh database), returns the hardcoded built-in default and writes it to the branch.

### Executor commands

```
RECIPE SET <json>                    -- set default recipe
RECIPE SET <name> <json>             -- set named recipe
RECIPE GET                           -- get default recipe
RECIPE GET <name>                    -- get named recipe
RECIPE LIST                          -- list all named recipes on this branch
```

**Estimated effort:** ~150 lines (engine + executor).

## 4. Recipe Types and Deserialization

### What it is

Rust types for the recipe schema defined in `retrieval-substrate.md`. Serde-deserializable from JSON. All fields `Option<T>` for the three-level merge.

### File

New: `crates/engine/src/search/recipe.rs`

### Types

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct Recipe {
    pub version: Option<u32>,
    pub retrieve: Option<RetrieveConfig>,
    pub expansion: Option<ExpansionConfig>,
    pub filter: Option<FilterConfig>,
    pub fusion: Option<FusionConfig>,
    pub rerank: Option<RerankConfig>,
    pub transform: Option<TransformConfig>,
    pub prompt: Option<String>,
    pub rag_context_hits: Option<usize>,
    pub rag_max_tokens: Option<usize>,
    pub models: Option<ModelsConfig>,
    pub version_config: Option<VersionOutputConfig>,
    pub control: Option<ControlConfig>,
}
```

All fields are `Option`. This enables three-level merge — each level only specifies the fields it overrides. A `merge(base: &Recipe, overlay: &Recipe) -> Recipe` function deep-merges overlay into base.

### Three-level merge implementation

```rust
impl Recipe {
    /// Deep merge: overlay wins where specified, base fills gaps.
    pub fn merge(base: &Recipe, overlay: &Recipe) -> Recipe {
        // For each field: use overlay if Some, else base
        // For nested structs: recurse
    }

    /// Resolve a recipe against the three levels.
    pub fn resolve(
        builtin: &Recipe,     // Level 1: hardcoded defaults
        branch: &Recipe,      // Level 2: stored on branch
        per_call: Option<&Recipe>, // Level 3: inline override
    ) -> Recipe {
        let merged = Recipe::merge(builtin, branch);
        match per_call {
            Some(override_recipe) => Recipe::merge(&merged, override_recipe),
            None => merged,
        }
    }
}
```

**Estimated effort:** ~300 lines (types + merge logic + tests).

## 5. Migrate Search Config from strata.toml

### What moves

| Config in strata.toml | Moves to | Recipe field |
|---|---|---|
| `bm25_k1` | Default recipe | `retrieve.bm25.k1` |
| `bm25_b` | Default recipe | `retrieve.bm25.b` |
| `auto_embed` | Stays in `db.configure()` | N/A (write-time, not query-time) |
| `embed_model` | Default recipe | `models.embed` |
| `embed_batch_size` | Stays in `db.configure()` | N/A (write-time) |

### Migration flow

On database open:
1. Read `strata.toml`
2. If `_system_` space has no `recipe:default`:
   - Build recipe from TOML values (bm25_k1, bm25_b, embed_model)
   - Write to `recipe:default` in `_system_` space
3. From now on, `strata.toml` search settings are ignored. Recipe is the source of truth.
4. Remove search settings from `strata.toml` template / documentation.

**Backward compatibility:** Existing databases with `strata.toml` settings get auto-migrated on first open. No data loss. The TOML settings become the initial recipe.

**Estimated effort:** ~100 lines (migration logic + TOML cleanup).

## 6. Refactor Existing Search Path

### Current state

The existing hybrid search orchestrator (`crates/search/src/hybrid.rs`) bypasses the Searchable trait for vector search. It calls `VectorStore::search_by_embedding()` directly. The orchestrator also handles expansion and reranking directly.

### What needs to change

1. **VectorStore Searchable:** Wire the trait to `search_by_embedding()` so the substrate can call it uniformly.
2. **JSON Searchable:** Wire to InvertedIndex for BM25 (currently only does field filtering).
3. **Event Searchable:** Wire to InvertedIndex for BM25 (currently returns empty).
4. **Remove orchestrator bypass:** The substrate replaces the orchestrator. But for v0.0, just ensure primitives work via the trait — the substrate itself is v0.1.

### Approach

Don't rewrite the existing orchestrator. Just fix the four primitives so they work through the Searchable trait. The existing orchestrator continues to function as-is (backward compat). The v0.1 substrate uses the same primitives through the same trait.

**Estimated effort:**
- VectorStore Searchable: ~80 lines
- JSON Searchable BM25: ~50 lines
- Event Searchable BM25: ~30 lines
- Total: ~160 lines

## 7. Move Shadow Collections to `_system_` Space

### Current state

Shadow embedding collections (`_system_embed_kv`, `_system_embed_json`, `_system_embed_event`) live on the `_system_` branch. All user branches share one set of embeddings.

### What changes

Move shadow collections to the `_system_` space on the user's branch.

**Benefits:**
- Embeddings fork with the branch via COW
- No cross-branch lookup for vector search
- Recipe's `vector.collections` resolves on the current branch
- Consistent with recipes, stats, and cache also living on the branch

### Migration

1. On database open, check if `_system_` branch has shadow collections
2. If yes, copy them to `_system_` space on the main branch
3. Update auto-embed hooks to write to `_system_` space on the current branch instead of `_system_` branch
4. Update vector search to look in `_system_` space on the current branch

**This is the largest change in v0.0.** It touches:
- Auto-embed write path (`crates/executor/src/handlers/embed_hook.rs`)
- Vector search read path (`crates/vector/src/store/search.rs`)
- Collection discovery (finding `_system_embed_*` collections)

**Estimated effort:** ~300 lines. Needs careful testing — embeddings are on the critical path for hybrid search.

## 8. Implementation Order

```
1. _system_ space on every branch (#2119)
   └─ Space validation, filtering, internal write API

2. Recipe types and deserialization
   └─ recipe.rs: types, merge logic, tests

3. Default recipe + set/get API
   └─ Store/retrieve recipes in _system_ space

4. Migrate search config from strata.toml
   └─ Auto-migration on database open

5. Fix primitive Searchable impls
   ├─ Event: wire to InvertedIndex (~30 lines)
   ├─ JSON: wire to InvertedIndex (~50 lines)
   └─ Vector: wire Searchable to search_by_embedding (~80 lines)

6. Move shadow collections to _system_ space
   └─ Auto-embed writes, vector search reads, migration

7. Executor commands (RECIPE SET/GET/LIST)
```

Steps 1-4 are foundation. Steps 5-6 are primitive readiness. Step 7 is API.

Steps 1-3 must be sequential (each depends on the previous). Steps 4-6 can run in parallel after step 3.

## 9. Testing Strategy

### 9.1 `_system_` space

- Write to `_system_` space via internal API → succeeds
- Write to `_system_` space via user API → rejected
- `db.kv.list()` → does not return `_system_` space entries
- Fork a branch → `_system_` space data inherited via COW
- Override `_system_` space data on fork → parent unaffected

- Default recipe exists on fresh database
- `set_recipe()` → `get_recipe()` round-trip

- Empty overlay → base recipe unchanged

### 9.3 Migration

- Database with strata.toml search settings → opens, migrates to recipe, BEIR scores unchanged
- Database without strata.toml search settings → opens with built-in defaults

### 9.4 Primitives

- JSON search("query") → returns BM25 hits (not empty)
- Event search("query") → returns BM25 hits (not empty)

- All existing search tests still pass

- Database with `_system_` branch shadow collections → migrated to `_system_` space on main

Same as v0.1: run nfcorpus + scifact + fiqa through both old path and new path. nDCG@10 must match within 0.001.
