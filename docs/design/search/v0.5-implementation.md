# v0.5 Implementation: Filter + Budget + Pagination

Builds on v0.4 (full retrieval stack). Adds post-retrieval filtering, budget enforcement, and pagination. These are the production-readiness features that make search usable in real applications.

**Prerequisite:** v0.4 landed and passing graph retrieval tests.

**Explicitly NOT in scope:** Scan (predicate-only retrieval), aggregation, group_by. These are query engine features — build them via DataFusion/Arrow integration when needed, not inside the search substrate. The recipe schema supports them as extension points but the implementation is deferred.

---

## 1. Scope

**In:**
- Post-retrieval predicate filtering (`filter` recipe section)
- Budget enforcement (`control.budget_ms`)
- Deduplication (`transform.deduplicate`)
- Pagination (`transform.offset`)
- Per-hit stage scores (`control.include_stage_scores`)

**Out:**
- `retrieve.scan` — pure predicate retrieval. Use primitive APIs (`db.kv.list()`, `db.json.list()`).
- `transform.aggregate` — faceted search. Use DataFusion over Arrow exports.
- `transform.group_by` — grouped results. Same.
- `transform.sort` by field — sort by score is the search default. Field sorting is a query concern.

## 2. Post-Retrieval Filtering

### What it does

After fusion produces a ranked list, filter removes candidates that don't match predicates. This is the "WHERE clause" of search.

```
BM25 + Vector → 100 candidates → RRF fusion → 87 unique → filter(year >= 2020) → 62 matching → limit 10
```

### Predicate evaluation

Predicates from the recipe:

```json
"filter": {
    "predicates": [
        {"field": "year", "op": "gte", "value": 2020},
        {"field": "status", "op": "in", "value": ["published", "preprint"]}
    ],
    "logic": "and",
    "time_range": {"start": "2025-01-01T00:00:00Z", "end": null}
}
```

For each candidate hit, the substrate must:
1. Resolve the entity_ref to the actual document (KV value, JSON document, event payload)
2. Extract the field value from the document
3. Evaluate the predicate against the field value
4. Keep or discard

### Field extraction per primitive

| Primitive | How to get field values |
|---|---|
| KV | Parse value as JSON, extract field. If value isn't JSON, only `_key` and `_type` fields available. |
| JSON | Direct field access on the JSON document. Supports nested paths (`metadata.author`). |
| Event | `event_type` and `timestamp` as built-in fields. Payload parsed as JSON for other fields. |
| Vector | Metadata JSON (top-level scalar fields only — matches existing MetadataFilter). |
| Graph | Node/edge properties (JSON). |

### Predicate operators

From `retrieval-substrate.md` section 3.6:

| Op | Implementation |
|---|---|
| `eq` | `field_value == predicate_value` |
| `neq` | `field_value != predicate_value` |
| `gt`, `gte`, `lt`, `lte` | Numeric or string comparison |
| `in` | `predicate_values.contains(field_value)` |
| `not_in` | `!predicate_values.contains(field_value)` |
| `contains` | Array field contains the value |
| `prefix` | String field starts with the value |
| `exists` | Field is present and non-null |

### Logic

- `"and"` (default): all predicates must match
- `"or"`: any predicate matches

### time_range

Special predicate on the entity's timestamp:
- KV: version timestamp
- JSON: `updated_at`
- Event: event timestamp
- Vector: `created_at`
- Graph: N/A (no timestamp on nodes/edges currently)

```rust
fn matches_time_range(hit: &SearchHit, range: &TimeRange) -> bool {
    let ts = resolve_timestamp(hit);
    ts >= range.start && range.end.map_or(true, |end| ts <= end)
}
```

### score_threshold

Discard hits below a minimum score:
```json
"filter": {"score_threshold": 0.1}
```

Applied after predicates. Eliminates low-relevance noise.

### Implementation

New: `crates/engine/src/search/filter.rs`

```rust
pub fn apply_filter(
    hits: &[SearchHit],
    filter: &FilterConfig,
    db: &Database,
    branch_id: &BranchId,
) -> StrataResult<Vec<SearchHit>> {
    hits.iter()
        .filter(|hit| {
            // score threshold
            if let Some(threshold) = filter.score_threshold {
                if hit.score < threshold { return false; }
            }
            // time range
            if let Some(ref range) = filter.time_range {
                if !matches_time_range(hit, range) { return false; }
            }
            // predicates
            if let Some(ref predicates) = filter.predicates {
                let doc = resolve_document(db, branch_id, &hit.doc_ref)?;
                match filter.logic.as_deref().unwrap_or("and") {
                    "and" => predicates.iter().all(|p| evaluate_predicate(p, &doc)),
                    "or" => predicates.iter().any(|p| evaluate_predicate(p, &doc)),
                    _ => true,
                }
            }
            true
        })
        .cloned()
        .collect()
}
```

**Estimated effort:** ~200 lines (filter.rs + predicate evaluator + field extraction per primitive).

## 3. Budget Enforcement

### What it does

Every search must complete within a time budget. If the budget is exceeded, return the best results found so far with `stats.budget_exhausted = true`.

### How it works

```rust
pub fn retrieve(db: &Database, request: &RetrievalRequest) -> StrataResult<RetrievalResponse> {
    let budget_ms = request.recipe.control
        .as_ref()
        .and_then(|c| c.budget_ms)
        .unwrap_or(5000);
    let deadline = Instant::now() + Duration::from_millis(budget_ms);

    // Step 1: Expand (check deadline)
    if Instant::now() > deadline { return partial_response(candidates, true); }

    // Step 2: Retrieve (check deadline between primitives)
    let bm25_hits = if Instant::now() < deadline { search_bm25(...) } else { vec![] };
    let vec_hits = if Instant::now() < deadline { search_vector(...) } else { vec![] };
    let graph_hits = if Instant::now() < deadline { search_graph(...) } else { vec![] };

    // Step 3: Fuse (fast, no check needed)
    // Step 4: Filter (check deadline)
    // Step 5: Rerank (skip entirely if over budget — most expensive step)

    let budget_exhausted = Instant::now() > deadline;
    // Return whatever we have
}
```

### Behavior

- The substrate checks the deadline between each major step
- If over budget after retrieval: skip reranking (the most expensive step) and return fused results
- If over budget during retrieval: return candidates from primitives that finished
- Never panic, never hang. Partial results always returned.
- `stats.budget_exhausted = true` signals to the caller that results may be incomplete

### Recipe parameter

```json
"control": {"budget_ms": 5000}
```

Default: 5000ms. Experimentable via `db.experiment()` (lower budget = faster but potentially worse results).

**Estimated effort:** ~60 lines. Deadline checks at each pipeline step.

## 4. Deduplication

### What it does

When multiple retrieval operators return the same entity (BM25 and vector both find doc:123), RRF already merges them. But dedup ensures no duplicates in the final output.

### Implementation

Already implicit in RRF (scores accumulate per entity_ref). Make it explicit:

```json
"transform": {"deduplicate": "entity_ref"}
```

After filtering, before limit:
```rust
let mut seen = HashSet::new();
hits.retain(|h| seen.insert(h.doc_ref.clone()));
```

**Estimated effort:** ~10 lines. Already mostly handled by RRF.

## 5. Pagination

### What it does

```json
"transform": {"limit": 10, "offset": 20}
```

Returns results 21-30. Standard cursor-based pagination.

### Implementation

After dedup:
```rust
let offset = transform.offset.unwrap_or(0);
let limit = transform.limit.unwrap_or(10);
hits = hits.into_iter().skip(offset).take(limit).collect();
```

**Estimated effort:** ~5 lines.

## 6. Per-Hit Stage Scores

### What it does

When `control.include_stage_scores` is true, each hit includes the score it received from each retrieval operator:

```json
{
    "entity_ref": {"type": "json", "key": "doc:123"},
    "score": 0.85,
    "rank": 1,
    "stage_scores": {
        "bm25": 0.72,
        "vector": 0.91,
        "graph": 0.15,
        "fusion": 0.85
    }
}
```

This tells agents and AutoResearch which operators are contributing value for each hit.

### Implementation

During RRF fusion, record per-source scores:

```rust
struct FusionCandidate {
    entity_ref: EntityRef,
    source_scores: HashMap<String, f32>,  // "bm25" → 0.72, "vector" → 0.91
    fused_score: f32,
}
```

When `include_stage_scores` is true, attach `source_scores` to each `SearchHit`.

**Estimated effort:** ~40 lines. Modify RRF to track per-source contributions.

## 7. Updated Pipeline

v0.5 changes from v0.4 in **bold**:

```
1. Resolve temporal context
2. Expand
3. Retrieve (BM25 + Vector + Graph)  **← budget: skip remaining if over deadline**
4. Fuse
5. **Filter (predicate evaluation, time_range, score_threshold)**
6. Rerank  **← budget: skip if over deadline**
7. If diff: retrieve at T1, compute diff
8. Enrich with version history
9. **Deduplicate**
10. **Paginate (offset + limit)**
11. Generate (if mode="rag")
12. Return **← budget_exhausted flag in stats**
```

## 8. Testing

### Filter tests
- Filter by single predicate → matching hits kept, non-matching removed
- Filter with `logic: "or"` → hits matching any predicate kept
- Filter with `score_threshold: 0.5` → low-score hits removed
- Filter with `time_range` → only hits within range kept
- Filter on JSON nested field (`metadata.author`) → works
- Filter on non-existent field with `exists: false` → matches docs without the field
- Empty filter → all hits pass through

### Budget tests
- Budget 5000ms with fast query → completes normally, budget_exhausted = false
- Budget 1ms → returns partial results, budget_exhausted = true
- Budget exceeded during retrieval → some primitives skipped, results from completed primitives returned
- Budget exceeded before reranking → reranking skipped, fusion scores used

### Pagination tests
- offset=0, limit=10 → first 10 results
- offset=10, limit=10 → results 11-20
- offset beyond result count → empty hits
- Deduplication before pagination → correct offset

## 10. What v0.5 Enables

After v0.5, the substrate is production-ready for search:

- Filtered search: "Find X where Y"
- Budget-bounded: no runaway queries
- Paginated: standard cursor-based navigation
- Observable: stage scores show which operators contribute value

What it does NOT enable:
- Pure predicate queries (scan) → use primitive APIs
- Aggregation/facets → use DataFusion over Arrow
- Group by → use DataFusion
- Sort by field → search sorts by score; field sorting is a query concern

These are deliberate omissions, not gaps. Search is ranked retrieval with filters. Query is DataFusion. Don't conflate them.
