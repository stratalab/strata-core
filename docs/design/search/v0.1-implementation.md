# v0.1 Implementation: Minimal Retrieval Substrate

The smallest thing that works. BM25 across all primitives, vector search, RRF fusion, top-k. One entry point.

---

## 1. Scope

**In:**
- BM25 keyword search across KV, JSON, Event, Graph via InvertedIndex
- Vector similarity search via HNSW
- RRF fusion of BM25 + vector results
- Top-k limiting
- Fixed response format (hits, answer, diff, aggregations, groups, stats)
- Recipe parsing (bm25, vector, fusion, transform.limit only)
- Single `retrieve()` entry point

**Out (v0.2+):**
- Scan (predicate-based retrieval)
- Filter (post-retrieval predicates)
- Expansion, reranking
- Aggregation, group_by, sort by field
- Graph-specific retrieval (PPR, neighborhood)
- Temporal (as_of, diff, version history)
- RAG generation
- Multi-model routing
- Budget enforcement
- Recipe storage / three-level merge

## 2. What Changes Per Primitive

### 2.1 KV — No changes

Already has a real BM25 implementation via InvertedIndex (`kv.rs:538-608`). Uses `score_top_k()`, resolves doc IDs, returns hits. Works today.

**One fix:** Pass `snapshot_version` from the substrate to ensure MVCC consistency. Currently ignored.

### 2.2 JSON — Wire to InvertedIndex

**Current:** `search()` only handles `field_filter` and `sort_by`. Returns empty if neither is provided. No keyword search.

**Change:** When `query` is non-empty and no `field_filter`/`sort_by`, fall back to InvertedIndex BM25 search. Same pattern as KV.

**File:** `crates/engine/src/primitives/json/mod.rs` (~line 1352)

**Approach:** Copy the KV pattern:
1. Get `InvertedIndex` from database extensions
2. Tokenize query
3. Call `index.score_top_k()` with branch filter
4. Filter results to `EntityRef::Json` only
5. Resolve doc IDs to JSON documents for snippets
6. Return `SearchResponse`

**Prerequisite:** JSON documents must be indexed in the InvertedIndex on write. Check if `json/mod.rs` calls `index.index_document()` during `set()`. If not, add it.

**Effort:** ~50 lines in search(), possibly ~20 lines in write path if indexing is missing.

### 2.3 Event — Wire to InvertedIndex

**Current:** `search()` returns empty. But `event_append()` already indexes text via `index.index_document()` (`event.rs:~371`).

**Change:** Implement `search()` using the same InvertedIndex pattern as KV.

**File:** `crates/engine/src/primitives/event.rs` (~line 993)

**Approach:** Same as KV:
1. Get `InvertedIndex`
2. Tokenize query
3. Call `index.score_top_k()`
4. Filter results to `EntityRef::Event` only
5. Resolve doc IDs to event payloads for snippets
6. Return `SearchResponse`

**Effort:** ~30 lines. The simplest change in v0.1.

### 2.4 Vector — Wire Searchable to search_by_embedding

**Current:** `Searchable::search()` returns empty for all modes. Real search lives in `search_by_embedding()` / `system_search_with_sources()`.

**Change:** When `mode == Vector || mode == Hybrid`, and a precomputed embedding is provided in the request, call the actual HNSW search and convert results to `SearchHit` format.

**File:** `crates/vector/src/store/mod.rs` (~line 680)

**Approach:**
1. Extract embedding from `SearchRequest.precomputed_embedding`
2. If present: call `self.search_by_embedding(branch_id, collection, embedding, k)`
3. Convert `VectorMatch` results to `SearchHit` (entity_ref, score, snippet from inline meta)
4. If no embedding: return empty (vector can't do keyword search)

**Question:** Which collections to search? For v0.1, search all `_system_embed_*` collections. The recipe's `vector.collections` field controls this in the full design, but v0.1 can hardcode the default.

**Effort:** ~80 lines. Main complexity is mapping VectorMatch → SearchHit and handling multiple collections.

### 2.5 Graph — No changes

Already has a real BM25 implementation via InvertedIndex (`graph/src/lib.rs:114-189`). Same pattern as KV. Works today.

Graph-specific retrieval (PPR, neighborhood) is out of scope for v0.1.

### 3.3 The retrieve function

```rust
// crates/engine/src/search/substrate.rs

pub fn retrieve(
    db: &Database,
    request: &RetrievalRequest,
) -> StrataResult<RetrievalResponse> {
    let start = Instant::now();
    let recipe = &request.recipe;
    let mut all_candidates: Vec<(&str, Vec<SearchHit>)> = Vec::new();
    let mut stages = HashMap::new();

    // Step 1: BM25 across all primitives
    if let Some(bm25_config) = &recipe.retrieve.bm25 {
        let bm25_start = Instant::now();
        let bm25_req = build_search_request(request, bm25_config);

        let mut bm25_hits = Vec::new();

        // Fan out to all primitives
        // KV
        if let Some(hits) = search_primitive(&db.kv, &bm25_req)? {
            bm25_hits.extend(hits);
        }
        // JSON
        if let Some(hits) = search_primitive(&db.json, &bm25_req)? {
            bm25_hits.extend(hits);
        }
        // Event
        if let Some(hits) = search_primitive(&db.event, &bm25_req)? {
            bm25_hits.extend(hits);
        }
        // Graph
        if let Some(hits) = search_primitive(&db.graph, &bm25_req)? {
            bm25_hits.extend(hits);
        }

        // Sort by score desc for RRF ranking
        bm25_hits.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap_or(Ordering::Equal));
        bm25_hits.truncate(bm25_config.k.unwrap_or(50));

        stages.insert("bm25".into(), StageStats {
            elapsed_ms: bm25_start.elapsed().as_secs_f64() * 1000.0,
            candidates: bm25_hits.len(),
        });
        all_candidates.push(("bm25", bm25_hits));
    }

    // Step 2: Vector search
    if let Some(vector_config) = &recipe.retrieve.vector {
        if let Some(embedding) = &request.embedding {
            let vec_start = Instant::now();
            let vec_hits = search_vector(db, request, vector_config, embedding)?;

            stages.insert("vector".into(), StageStats {
                elapsed_ms: vec_start.elapsed().as_secs_f64() * 1000.0,
                candidates: vec_hits.len(),
            });
            all_candidates.push(("vector", vec_hits));
        }
    }

    // Step 3: RRF fusion
    let fusion_config = recipe.fusion.as_ref();
    let fused = rrf_fuse(&all_candidates, fusion_config);

    stages.insert("fusion".into(), StageStats {
        elapsed_ms: 0.0, // trivial
        candidates: fused.len(),
    });

    // Step 4: Limit
    let limit = recipe.transform
        .as_ref()
        .and_then(|t| t.limit)
        .unwrap_or(10);
    let hits: Vec<SearchHit> = fused.into_iter().take(limit).collect();

    // Step 5: Return fixed format
    Ok(RetrievalResponse {
        hits,
        answer: None,
        diff: None,
        aggregations: None,
        groups: None,
        stats: RetrievalStats {
            snapshot_version: db.current_version(),
            recipe_used: "inline".into(),
            elapsed_ms: start.elapsed().as_secs_f64() * 1000.0,
            stages,
        },
    })
}
```

### 3.4 RRF fusion

```rust
fn rrf_fuse(
    sources: &[(&str, Vec<SearchHit>)],
    config: Option<&FusionConfig>,
) -> Vec<SearchHit> {
    let k = config.and_then(|c| c.k).unwrap_or(60) as f32;
    let weights = config.and_then(|c| c.weights.as_ref());

    // entity_ref → accumulated RRF score
    let mut scores: HashMap<EntityRef, f32> = HashMap::new();
    let mut best_hit: HashMap<EntityRef, SearchHit> = HashMap::new();

    for (source_name, hits) in sources {
        let weight = weights
            .and_then(|w| w.get(*source_name))
            .copied()
            .unwrap_or(1.0);

        for (rank, hit) in hits.iter().enumerate() {
            let rrf_score = weight / (k + rank as f32 + 1.0);
            *scores.entry(hit.doc_ref.clone()).or_default() += rrf_score;
            best_hit.entry(hit.doc_ref.clone()).or_insert_with(|| hit.clone());
        }
    }

    // Sort by RRF score desc, then entity_ref asc for determinism
    let mut results: Vec<SearchHit> = scores.iter().map(|(entity, &score)| {
        let mut hit = best_hit[entity].clone();
        hit.score = score;
        hit
    }).collect();

    results.sort_by(|a, b| {
        b.score.partial_cmp(&a.score)
            .unwrap_or(Ordering::Equal)
            .then_with(|| a.doc_ref.cmp(&b.doc_ref))
    });

    // Assign ranks
    for (i, hit) in results.iter_mut().enumerate() {
        hit.rank = (i + 1) as u32;
    }

    results
}
```

## 4. Executor Wiring

The executor's search handler currently dispatches to the hybrid search orchestrator. For v0.1, it should dispatch to the substrate instead.

**File:** `crates/executor/src/handlers/search.rs`

**Change:** When a recipe is provided (or the default recipe is used), call `substrate::retrieve()` instead of the existing search path.

For v0.1, the existing search path can remain as a fallback. The substrate is an alternative entry point, not a replacement yet. This allows incremental testing.

```rust
// In the search handler:
if let Some(recipe) = parse_recipe(request) {
    // New path: substrate
    substrate::retrieve(db, &RetrievalRequest {
        query: request.query,
        recipe,
        branch_id: request.branch_id,
        embedding: request.precomputed_embedding,
    })
} else {
    // Old path: existing orchestrator (backward compatible)
    hybrid_search(db, request)
}
```

## 6. Testing Strategy

### 6.1 Unit tests

- Recipe deserialization: empty recipe, bm25-only, vector-only, hybrid, all defaults
- RRF fusion: single source, two sources, three sources, weighted, empty lists, tie-breaking determinism
- Substrate retrieve: bm25 only, vector only, hybrid, limit=0, limit=1

### 6.2 Integration tests

- Insert data across KV, JSON, Event → search via substrate → verify all primitives return results
- Insert data + embeddings → hybrid search → verify fusion merges correctly
- Same query, same recipe, two calls → identical results (determinism invariant)
- Recipe with only bm25 → no vector candidates
- Recipe with only vector → no bm25 candidates

### 6.3 BEIR regression (the real test)

The substrate must produce **identical nDCG@10 scores** to the existing search path for equivalent configurations. This is the regression test that proves v0.1 is correct.

**Harness:** strata-eval (~/Documents/GitHub/strata-eval/)
- 15 BEIR datasets available, Pyserini baselines for comparison
- Existing results: hybrid nDCG@10 = 0.4818 average across 14 datasets

**Fast validation (5 minutes):**
```bash
cd ~/Documents/GitHub/strata-eval
python run.py beir --dataset nfcorpus scifact fiqa --mode keyword hybrid
```

**Regression check:**
1. Run the fast subset through the **existing** search path → capture nDCG@10 per dataset
2. Run the same subset through the **substrate** path → capture nDCG@10 per dataset
3. Compare. If they match (within floating-point tolerance), the substrate is correct.
4. If they don't match, the substrate has a bug. Diff the result lists to find where.

**Success criteria:** nDCG@10 difference < 0.001 across all three datasets for both keyword and hybrid modes.

### 6.4 Existing test regression

All existing search tests continue to pass via the old path. The substrate is an additive code path, not a replacement. The old path remains as fallback until the substrate is proven equivalent.

## 7. What v0.1 Enables

Once this lands:

- `db.search("query")` goes through a single substrate entry point
- The recipe controls what runs (bm25, vector, or both)
- Results come back in the fixed 6-field format
- `db.experiment()` can compare recipes (different k1/b values, different fusion weights)
- AutoResearch has something to optimize

What it does NOT enable yet:
- No scan/filter/aggregate/group_by (v0.2)
- No expansion/reranking (v0.2)
- No RAG/temporal (v0.3)
- No recipe storage/merge (v0.2)
- No budget enforcement (v0.2)

---

## 8. Success Criteria

1. `retrieve(db, query, recipe)` returns results from all four BM25-capable primitives
2. Vector search works when embedding is provided
3. RRF fusion produces deterministic ranked results
4. The fixed response format is correct (all 6 fields present)
5. Existing search tests continue to pass
6. A simple `db.experiment()` comparing two recipes returns the winner
