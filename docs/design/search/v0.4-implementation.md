# v0.4 Implementation: Graph-Augmented Retrieval

Builds on v0.3 (substrate + expansion + reranking + temporal). Adds the graph as a retrieval signal — the key differentiator that no other embedded database has.

**Prerequisite:** v0.3 landed and passing temporal tests.

---

## 1. Why PPR

We evaluated seven approaches. Personalized PageRank is the clear winner for Strata:

| Approach | Effort | Value | Practicality |
|---|---|---|---|
| **Personalized PageRank (PPR)** | **~50 lines on existing PageRank** | **Very high (multi-hop)** | **P0** |
| Typed traversal | Primitives exist, need query planning | High (domain-specific) | P1 |
| Neighborhood expansion (GraphBoost) | Already built | Medium | Already in codebase |
| LazyGraphRAG communities | CDLP exists, need co-occurrence graph | High (global queries) | Future |
| Shortest path scoring | SSSP exists, need wiring | Medium | Future |
| Bipartite matching | entity_ref exists | Medium-low | Future |
| GNN approaches | Requires training pipeline | Low (not embedded-friendly) | Backlog |

PPR is the HippoRAG algorithm (NeurIPS 2024). It achieves 72.9% Recall@5 vs 65.6% for ColBERTv2 on multi-hop benchmarks. The key: it finds entities that are structurally related to the query across multiple hops, in a single pass.

Strata's existing `pagerank_with_index` needs one modification: accept a personalization vector instead of uniform initialization. Everything else — adjacency index, convergence logic, dangling node handling — stays the same.

## 2. Scope

**In:**
- Personalized PageRank implementation (~50 lines on existing PageRank)
- Anchor node resolution (query text → graph nodes)
- PPR as a retrieval operator in the substrate
- Typed traversal via ontology edge-type constraints
- RRF fusion of graph + BM25 + vector results
- Recipe integration (`retrieve.graph` section)

**Out (v0.5+):**
- LazyGraphRAG-style community queries
- GNN-based retrieval
- Automatic entity extraction at write time (user builds the graph)
- Graph-specific temporal (bi-temporal edges)

## 3. How It Works

### 3.1 End-to-end flow

```
User query: "What drugs interact with metformin?"
                │
                ▼
1. ANCHOR NODE RESOLUTION
   Tokenize query → ["drug", "interact", "metformin"]
   Match against node labels in graph → [node:metformin]
                │
                ▼
2. PERSONALIZED PAGERANK
   Seed: {metformin: 1.0}
   Damping: 0.5 (from recipe)
   Edge type filter: ontology-guided (optional)
   Result: ranked nodes by PPR score
     node:warfarin       0.23
     node:insulin        0.18
     node:lactic_acidosis 0.12
     node:type_2_diabetes 0.09
     ...
                │
                ▼
3. NODE → DOCUMENT RESOLUTION
   For each high-PPR node, follow entity_ref to source documents:
     node:warfarin → doc:warfarin-interactions
     node:insulin → doc:insulin-guide
     node:lactic_acidosis → doc:metformin-adverse
                │
                ▼
4. SCORED CANDIDATES
   Score = PPR score of the node that links to the document
   If multiple nodes link to the same document, take max PPR score
     doc:warfarin-interactions   0.23
     doc:insulin-guide           0.18
     doc:metformin-adverse       0.12
                │
                ▼
5. RRF FUSION (with BM25 and vector candidates)
   Graph weight from recipe (default 0.3)
```

### 3.2 Anchor node resolution

The substrate needs to go from query text to graph seed nodes. Two strategies, both deterministic (no model needed):

**Strategy A: Token matching (default)**
1. Tokenize query using the recipe's BM25 tokenizer (stemmer, stopwords)
2. For each token, check if a node with that label exists in the named graph
3. Matching nodes become PPR seeds with equal weight

**Strategy B: Prefix matching**
1. Same tokenization
2. Match tokens as prefixes against node labels (catches "metformin" matching "metformin_hydrochloride")

Both are deterministic, O(tokens × graph_nodes) with a hash lookup. No LLM needed.

**Recipe parameter:**
```json
"graph": {
    "anchor_match": "token"
}
```
Options: `"token"` (exact stemmed match), `"prefix"` (prefix match).

### 3.3 Personalized PageRank

**Current code:** `crates/graph/src/analytics.rs` — `pagerank_with_index()`

**Modification:**

```rust
pub struct PageRankOptions {
    pub damping: f64,           // existing
    pub max_iterations: usize,  // existing
    pub tolerance: f64,         // existing
    pub personalization: Option<HashMap<String, f64>>,  // NEW
}
```

When `personalization` is `Some`:
- Initialize ranks: `rank[node] = personalization.get(node).unwrap_or(0.0)` (instead of `1.0/n`)
- Teleport: `base[node] = (1.0 - d) * personalization.get(node).unwrap_or(0.0)` (instead of `(1.0-d)/n`)
- Everything else identical: adjacency traversal, dangling node redistribution, convergence check

**Damping factor:**
- Standard PageRank uses 0.85 (explore far from seeds, 15% restart)
- HippoRAG uses 0.5 (explore less, restart more often — stays closer to seeds)
- Recipe parameter: `graph.damping` (default 0.5)

Lower damping = results cluster around query entities. Higher damping = results spread further through the graph. AutoResearch can tune this.

### 3.4 Typed traversal

The ontology constrains which edges PPR follows. This is critical for precision — "treats" edges are relevant for a medical query, "manufactured_by" edges are not.

**Implementation:** Filter the adjacency index before running PPR.

```rust
fn build_filtered_adjacency(
    graph: &GraphData,
    edge_types: Option<&[String]>,
) -> AdjacencyIndex {
    // If edge_types specified, only include matching edges
    // Otherwise, include all edges
}
```

The edge type filter is already supported by Strata's BFS (`BfsOptions::edge_types`). Apply the same filter to PPR's adjacency traversal.

**Recipe parameter:**
```json
"graph": {
    "edge_types": ["treats", "causes", "interacts_with"]
}
```

If omitted, all edge types are used. If specified, PPR only propagates along those edge types.

**Ontology integration:** The user defines the ontology. The recipe can reference edge types from the ontology. AutoResearch can experiment with different edge type subsets.

### 3.5 Node → document resolution

PPR produces ranked graph nodes. The substrate needs to convert these to document references for fusion with BM25 and vector results.

**Existing infrastructure:** `entity_ref` on graph nodes already links nodes to KV/JSON documents. The `GraphBoost` module in `boost.rs` already does this mapping.

**Implementation:**
```rust
fn resolve_nodes_to_documents(
    graph: &GraphData,
    ppr_scores: &HashMap<String, f64>,
    k: usize,
) -> Vec<SearchHit> {
    let mut doc_scores: HashMap<EntityRef, f64> = HashMap::new();

    for (node_id, &score) in ppr_scores.iter() {
        if let Some(node) = graph.get_node(node_id) {
            if let Some(ref entity_ref) = node.entity_ref {
                let entry = doc_scores.entry(entity_ref.clone()).or_default();
                *entry = entry.max(score); // max PPR score across nodes linking to same doc
            }
        }
    }

    // Sort by score, take top k, convert to SearchHit
    let mut hits: Vec<_> = doc_scores.into_iter()
        .map(|(doc_ref, score)| SearchHit {
            doc_ref,
            score: score as f32,
            rank: 0,
            snippet: None,
        })
        .collect();

    hits.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap_or(Ordering::Equal));
    hits.truncate(k);
    hits
}
```

## 4. Recipe Integration

The `retrieve.graph` section controls graph retrieval:

```json
{
    "retrieve": {
        "bm25": {},
        "vector": {},
        "graph": {
            "graph": "medical_ontology",
            "strategy": "ppr",
            "k": 50,
            "damping": 0.5,
            "max_hops": 2,
            "max_neighbors_per_hop": 50,
            "anchor_match": "token",
            "edge_types": ["treats", "causes", "interacts_with"]
        }
    },
    "fusion": {
        "weights": {"bm25": 1.0, "vector": 1.0, "graph": 0.3}
    }
}
```

All parameters from `retrieval-substrate.md` section 3.5:

| Recipe path | Default | What `db.experiment()` can tune |
|---|---|---|
| `graph.graph` | Required | Which user-defined graph to use |
| `graph.strategy` | `"ppr"` | PPR vs neighborhood (future strategies) |
| `graph.k` | 50 | How many graph candidates to produce |
| `graph.damping` | 0.5 | PPR restart probability (lower = closer to seeds) |
| `graph.max_hops` | 2 | For neighborhood strategy |
| `graph.max_neighbors_per_hop` | 50 | For neighborhood strategy |
| `graph.anchor_match` | `"token"` | How to resolve query → seed nodes |
| `graph.edge_types` | All | Ontology-constrained traversal |
| `fusion.weights.graph` | 0.3 | Graph signal weight in RRF |

### Experimental recipes

```python
result = db.experiment(
    recipes={
        "no_graph":       {"retrieve": {"bm25": {}, "vector": {}}},
        "graph_low":      {"retrieve": {"bm25": {}, "vector": {}, "graph": {"graph": "medical", "damping": 0.3}}, "fusion": {"weights": {"graph": 0.1}}},
        "graph_medium":   {"retrieve": {"bm25": {}, "vector": {}, "graph": {"graph": "medical", "damping": 0.5}}, "fusion": {"weights": {"graph": 0.3}}},
        "graph_high":     {"retrieve": {"bm25": {}, "vector": {}, "graph": {"graph": "medical", "damping": 0.7}}, "fusion": {"weights": {"graph": 0.5}}},
        "typed_only":     {"retrieve": {"bm25": {}, "vector": {}, "graph": {"graph": "medical", "edge_types": ["treats"]}}, "fusion": {"weights": {"graph": 0.3}}},
    },
    eval_set=eval_set
)
```

Five recipes, each one line of delta. AutoResearch discovers the optimal damping and weight for this dataset.

## 5. Updated Pipeline

v0.4 changes from v0.3 in **bold**:

```
1. Resolve temporal context (as_of, diff)

2. Expand (if expansion section present)

3. Retrieve (parallel)
   ├─ BM25 (original + lex expansions)
   ├─ Vector (original + vec/hyde embeddings)
   └─ **Graph (PPR from anchor nodes, typed traversal)**

4. Fuse → single ranked list (graph weighted per recipe)

5. Rerank

6. If diff: retrieve at T1, compute diff

7. Enrich with version history

8. Transform → limit

9. Generate (if mode="rag")

10. Return
```

## 7. Testing Strategy

### 7.1 PPR correctness

- Single seed node → PPR concentrates around that node's neighborhood
- Two seed nodes → PPR balances between both neighborhoods
- Damping 0.1 (high restart) → results very close to seeds
- Damping 0.9 (low restart) → results spread across graph
- Disconnected graph → PPR stays in seed's component
- Empty graph → zero candidates, no crash

### 7.2 Anchor resolution

- Query "metformin" + graph has node "metformin" → anchor found
- Query "metformin dosage" + graph has node "metformin" → "metformin" anchored, "dosage" not
- No matching nodes → graph operator returns zero candidates, BM25/vector still work
- Prefix matching: "metform" matches "metformin" node

### 7.3 Typed traversal

- Edge types ["treats"] → PPR only follows "treats" edges
- Edge types [] → PPR follows nothing, returns only seed nodes
- Edge types omitted → PPR follows all edges

### 7.4 Fusion

- Graph candidates fuse with BM25 + vector via RRF
- Graph weight 0.0 → graph has no effect
- Graph weight 1.0 → graph fully participates
- Document found by all three (BM25 + vector + graph) → boosted by appearing in all three lists

### 7.5 Multi-hop benchmark

The real test. Create a graph with known multi-hop paths:
```
metformin -[treats]→ diabetes -[associated_with]→ gene_X -[expressed_in]→ kidney
```
Query: "kidney gene expression related to metformin"
- BM25 alone: might find "metformin" documents but NOT "kidney" documents
- Vector alone: might find "kidney" documents but weak connection to "metformin"
- PPR: starts at "metformin", follows edges to "kidney" via "diabetes" and "gene_X", finds documents linked to "kidney"
- Fused: PPR adds the missing multi-hop results that neither BM25 nor vector found

### 7.6 BEIR + multi-hop

- Standard BEIR datasets: graph should not hurt (weight is low, graceful when no graph exists)
- HotpotQA / MuSiQue: graph should help significantly (multi-hop queries)
- This is the ablation study L2 from `search-strategy.md`

## 10. What v0.4 Enables

After v0.4, Strata has the complete retrieval stack:

```
BM25 (keywords) + Vector (semantics) + Graph (structure) + Expansion + Reranking + Temporal
```

Fused via RRF. Every parameter in the recipe. Experimentable via `db.experiment()`.

This is the ablation study from `search-strategy.md`:
- L0: BM25 ✓ (v0.1)
- L1: BM25 + Vector ✓ (v0.1)
- L2: BM25 + Vector + Graph ✓ (v0.4)
- L3: + Expansion + Reranking ✓ (v0.2)
- L4: AutoResearch-tuned (requires `db.experiment()` formalization)

The hypothesis to test: ontology-guided PPR adds retrieval signal that is orthogonal to both text and embedding similarity, particularly on multi-hop queries.
