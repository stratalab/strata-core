# v0.3 Implementation: Temporal Search

Builds on v0.2 (substrate + expansion + reranking). Adds temporal capabilities that are uniquely Strata. The infrastructure already exists — this is mostly wiring.

**Prerequisite:** v0.2 landed and passing BEIR ablation.

---

## 1. Scope

**In:**
- Point-in-time search (`as_of`)
- Temporal diff (`diff`)
- Per-hit version history (`version` recipe section)
- Temporal RAG (diff + mode="rag") — the intelligence layer generates explanations of what changed
- `snapshot_version` enforcement across all primitives

**Out (v0.4+):**
- Scan, filter, aggregate, group_by
- Graph-specific retrieval (PPR, neighborhood)
- Recipe storage / three-level merge from `_system_` space (use inline recipes until then)
- `db.experiment()` formal API

## 3. Three Temporal Features

### 3.1 Point-in-time search (`as_of`)

**User sees:**
```python
results = db.search("metformin side effects", as_of="2025-01-01")
```

**What happens:**
1. Resolve timestamp "2025-01-01" → MVCC snapshot version via `time_range()` lookup
2. Set `snapshot_version` on the retrieval request
3. Substrate executes recipe as normal, but every primitive reads at that snapshot
4. BM25: `MvccIterator` filters to `commit_id ≤ snapshot_version` (already works, just needs to use the version)
5. Vector: `search_at(query, k, as_of_ts)` filters HNSW nodes by `created_at ≤ as_of_ts` (already works)
6. Results reflect the database state at that point in time

**Implementation:**
- **Timestamp → snapshot version:** New helper in `Database`: `version_at_timestamp(ts: u64) -> Option<u64>`. Binary search through version/timestamp mapping.
- **Pass snapshot_version to primitives:** The substrate sets `SearchRequest.snapshot_version`. Each primitive's `search()` must respect it.
- **BM25 change:** In KV/JSON/Event/Graph search — pass `max_version` to InvertedIndex queries. Currently the index doesn't filter by version. Options:
  - (a) InvertedIndex gains version-aware scoring (filter postings by version)
  - (b) Post-filter: BM25 returns candidates, then resolve each via `get_at_timestamp()` — discard if not visible at that snapshot
  - Option (b) is simpler for v0.3. Option (a) is faster but requires index changes.

**Effort:** ~100 lines. Mostly passing snapshot_version through and adding post-filter.

### 3.2 Temporal diff (`diff`)

**User sees:**
```python
results = db.search("metformin side effects", diff=("2024-01-01", "2025-06-01"))
```

**What happens:**
1. Run the recipe at snapshot T1 → `results_before`
2. Run the recipe at snapshot T2 → `results_after`
3. Compute diff:
   - `new_hits`: in `results_after` but not in `results_before` (by entity_ref)
   - `removed_hits`: in `results_before` but not in `results_after`
   - `changed_hits`: in both but score changed by more than a threshold (default 0.05)
4. Return both `hits` (results at T2) and `diff`

**Implementation:**
```rust
fn compute_diff(before: &[SearchHit], after: &[SearchHit]) -> DiffResponse {
    let before_set: HashMap<EntityRef, &SearchHit> = before.iter().map(|h| (h.doc_ref.clone(), h)).collect();
    let after_set: HashMap<EntityRef, &SearchHit> = after.iter().map(|h| (h.doc_ref.clone(), h)).collect();

    let new_hits: Vec<_> = after.iter()
        .filter(|h| !before_set.contains_key(&h.doc_ref))
        .collect();

    let removed_hits: Vec<_> = before.iter()
        .filter(|h| !after_set.contains_key(&h.doc_ref))
        .collect();

    let changed_hits: Vec<_> = after.iter()
        .filter_map(|h| {
            before_set.get(&h.doc_ref).and_then(|prev| {
                if (h.score - prev.score).abs() > 0.05 {
                    Some(ChangedHit { hit: h, previous_score: prev.score, previous_rank: prev.rank })
                } else {
                    None
                }
            })
        })
        .collect();

    DiffResponse { before_snapshot, after_snapshot, new_hits, removed_hits, changed_hits }
}
```

**Effort:** ~80 lines. The substrate calls `retrieve()` twice (different snapshot versions) and diffs.

### 3.3 Per-hit version history (`version` recipe section)

**User sees:**
```python
# Recipe includes version config
db.set_recipe({
    "retrieve": {"bm25": {}, "vector": {}},
    "version": {"include_history": true, "depth": 3}
})

results = db.search("metformin dosage")
# Each hit includes versions array
```

**What happens:**

After the substrate produces hits, for each hit with a KV or JSON entity_ref:
1. Call `db.kv.getv(key)` or `db.json.getv(doc_id)`
2. Take the first `depth` versions
3. Extract snippet from each version
4. Attach to the hit

**Implementation:**
```rust
fn enrich_with_versions(db: &Database, hits: &mut [SearchHit], config: &VersionOutputConfig) {
    if !config.include_history { return; }
    let depth = config.depth.unwrap_or(3);

    for hit in hits.iter_mut() {
        match &hit.doc_ref {
            EntityRef::Kv { branch_id, key, .. } => {
                if let Ok(Some(history)) = db.kv.getv(branch_id, "default", key) {
                    hit.versions = history.versions().iter()
                        .take(depth)
                        .map(|v| VersionInfo {
                            version: v.version(),
                            timestamp: v.timestamp(),
                            snippet: extract_snippet(v.value()),
                        })
                        .collect();
                }
            },
            EntityRef::Json { branch_id, doc_id, .. } => {
                if let Ok(Some(history)) = db.json.getv(branch_id, "default", doc_id) {
                    hit.versions = history.versions().iter()
                        .take(depth)
                        .map(|v| VersionInfo {
                            version: v.version(),
                            timestamp: v.timestamp(),
                            snippet: extract_snippet_json(v.value()),
                        })
                        .collect();
                }
            },
            _ => {} // Events and graph nodes don't have getv()
        }
    }
}
```

**Effort:** ~80 lines. Straightforward — call getv(), truncate, attach.

## 4. Temporal RAG

When `mode="rag"` and `diff` is provided, the intelligence layer gets a different prompt:

**Standard RAG prompt:**
```
Context:
[1] {snippet}
[2] {snippet}

Question: {query}
```

**Temporal RAG prompt:**
```
Context (current):
[1] {snippet from T2 results}
[2] {snippet from T2 results}

Changes since {T1}:
NEW: [3] {snippet from new hit}
REMOVED: {snippet from removed hit} (no longer in results)
CHANGED: [1] moved from rank #4 to rank #1

Question: {query}
How has the information changed between {T1} and {T2}?
```

The RAG model sees the diff context and explains what changed.

**Effort:** ~50 lines. Prompt construction variant in the intelligence layer.

## 6. Updated Pipeline

v0.3 changes from v0.2 in **bold**:

```
1. **Resolve temporal context**
   ├─ as_of → resolve timestamp to snapshot_version
   └─ diff → resolve both timestamps to snapshot_versions

2. Expand (if expansion section present)

3. Retrieve (at snapshot_version if as_of, or at T2 if diff)

4. Fuse

5. Rerank

6. **If diff: retrieve again at T1, compute diff**

7. **Enrich with version history (if version.include_history)**

8. Transform → limit

9. Generate (if mode="rag", with temporal prompt if diff)

10. Return { hits, answer, diff, aggregations, groups, stats }
```

## 7. Testing Strategy

### 7.1 Point-in-time tests

- Insert doc A at T1, doc B at T2. Search with `as_of=T1` → only doc A found.
- Insert doc, update it at T2. Search with `as_of=T1` → original version returned.
- Vector search with `as_of` → HNSW filters by created_at.

### 7.2 Diff tests

- Insert docs at T1, insert more at T2. Diff(T1, T2) → new_hits contains T2 docs.
- Delete doc between T1 and T2. Diff → removed_hits contains deleted doc.
- Update doc between T1 and T2. Diff → changed_hits if score changed.
- Diff with identical snapshots → no new, removed, or changed hits.

### 7.3 Version history tests

- Insert doc, update 5 times. Search with `version.depth=3` → 3 versions per hit.
- Version history ordered newest-first (version[0] = latest).
- Hit without version history (Event, Graph) → empty versions array.

### 7.4 Determinism

- Same query + same recipe + same as_of → identical results.
- Diff(T1, T2) produces the same diff every time.

## 8. Success Criteria

1. `as_of` search returns results reflecting historical database state
2. `diff` search correctly identifies new, removed, and changed hits
3. Per-hit version history shows document evolution
4. Temporal RAG explains what changed (when intelligence layer is available)
5. All v0.2 tests still pass
6. `snapshot_version` is respected by all primitives (not ignored)
