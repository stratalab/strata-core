# AI agent memory architectures: a 2024–2026 survey

**No existing database fully serves what AI agents need for memory.** Every production agent memory system today either stitches together 3–5 databases (vector, graph, relational, cache) or pushes critical memory management—decay, consolidation, conflict resolution—into application code or LLM inference calls. This comprehensive survey of 11 systems reveals a field converging on similar requirements but diverging wildly in implementation. The central architectural tension: agent memory demands unified multi-modal retrieval (vector similarity + graph traversal + temporal filtering + keyword search) in a single transaction, but no production database delivers this natively. The race to close this gap is underway, with multi-model databases like SurrealDB 3.0 and memory middleware like Mem0, Zep, and Letta competing to become the default memory infrastructure for AI agents.

## The landscape at a glance

The table below captures the core technical properties of all 11 systems surveyed. Three patterns emerge: most systems use **PostgreSQL or vector databases** as storage backends, **none implement robust automatic forgetting**, and only **one** (LangGraph) offers true memory-state branching.

|---|---|---|---|---|---|
| **MemGPT / Letta** | PostgreSQL + pgvector | Tiered hierarchy (core/recall/archival) | ❌ None (FIFO eviction only) | ❌ No | Embedding similarity + string matching |
| **CoALA** | Framework-agnostic (conceptual) | Modular typed (working/episodic/semantic/procedural) | ❌ Not addressed | ❌ Not addressed | Flexible: rule-based, sparse, dense, or hybrid |

## Academic architectures that defined the field

Four research systems established the conceptual vocabulary that every production platform now builds on. Stanford's Generative Agents introduced the **recency × relevance × importance** scoring formula that became the de facto retrieval paradigm. Princeton's CoALA provided the cognitive taxonomy. A-MEM and MaRS pushed toward graph structures and forgetting.

**Generative Agents** (Park et al., Stanford, UIST 2023) remains the most influential memory architecture in the field. Its memory stream stores observations, reflections, and plans as a chronological append-only list. The retrieval formula—`score = α_recency × recency + α_importance × importance + α_relevance × relevance`—with all α weights set to **1** and min-max normalization, combines three signals: exponential recency decay at **γ = 0.995 per hour** (a memory's recency score drops to ~0.03 after one month), LLM-rated importance on a 1–10 poignancy scale, and cosine similarity of OpenAI `text-embedding-ada-002` embeddings. The reflection mechanism triggers when cumulative importance scores exceed a threshold, generating ~2–3 reflections per simulated day. These create a **reflection tree** layering abstract insights over raw observations. Storage is bare-bones research infrastructure—Python lists serialized to JSON files—but the scoring formula has been adopted or adapted by virtually every subsequent system.

**CoALA** (Sumers, Yao, Narasimhan, Griffiths; Princeton, TMLR 2024) is a conceptual framework, not an implementation. It borrows from cognitive architectures (Soar, ACT-R) to propose four memory types: **working memory** (the LLM's current context), **episodic memory** (sequences of past experiences), **semantic memory** (factual knowledge), and **procedural memory** (skills and rules, split into implicit LLM weights and explicit code/prompts). Its key contribution is the decision cycle—retrieve → reason → plan → execute → learn—that formalizes how information flows between working and long-term memory. CoALA prescribes no storage backends, retrieval algorithms, or forgetting mechanisms, which limits its practical utility but makes it a useful organizing lens.

**A-MEM** (Xu et al., Rutgers/AIOS Foundation, arXiv:2502.12110, accepted NeurIPS 2025) adapts the **Zettelkasten** personal knowledge management method for agents. Each memory becomes a structured "note" with raw content, LLM-generated keywords/tags, a contextual description, and a dense embedding vector. The system's three modules—note construction, link generation, and memory evolution—create a dynamic knowledge graph where notes link to semantically related neighbors. Memory evolution is the key innovation: when new memories arrive, existing notes' representations can be **updated in place**, enabling continuous refinement rather than simple accumulation. Retrieval combines embedding similarity search (via ChromaDB in the reference implementation) with graph traversal through linked "boxes" of notes. On DialSim benchmarks, A-MEM achieves **F1=3.45** versus MemGPT's 1.18, with sub-10 microsecond retrieval at 1M notes.

**MaRS** (Alqithami, arXiv:2512.12856, December 2025) is the only academic system that treats forgetting as a **first-class architectural primitive**. Its memory nodes are typed tuples containing content, timestamp, type (episodic/semantic/social/task), sensitivity score, token weight, and provenance. The graph structure uses four edge types: temporal, semantic, causal/provenance, and social. Six forgetting policies—FIFO, LRU, Priority Decay, Reflection-Summary, Random-Drop, and Hybrid—are triggered when total token weight exceeds configurable budgets. The Hybrid policy (staging temporal → importance → privacy signals) achieves **~0.911 composite score** across 300 simulation runs. MaRS is also the only system with a built-in privacy engine supporting (ε, δ)-differential privacy and GDPR-aligned sensitivity weighting. The limitation: no public code repository exists despite detailed theoretical specification.

## Production platforms competing for the memory layer

Three open-source platforms—Letta, Mem0, and Zep—have emerged as the primary contenders for production agent memory, each with a fundamentally different architectural philosophy.

**MemGPT / Letta** treats the LLM context window as RAM and external storage as disk, implementing **virtual context management** inspired by operating system memory paging. The three tiers are core memory (labeled blocks always in the system prompt, editable via tool calls like `core_memory_append`), recall memory (complete conversation history with string-matching search), and archival memory (long-term vector store with embedding-based search). Storage is **PostgreSQL with pgvector**, standardized after earlier ChromaDB support. The agent autonomously manages its own memory through designated tool calls—deciding what to store, retrieve, and update. When context fills to ~70%, a warning triggers the agent to offload important information. Letta raised **$10M** (Felicis-led, September 2024) to commercialize this as an agent framework. The V1 architecture (2025) deprecates the original "heartbeat" mechanism in favor of native reasoning. Key gap: recall memory uses basic string matching rather than semantic search, and there is no forgetting mechanism—memory grows unboundedly.

**Mem0** provides the simplest integration path—a universal memory layer that extracts facts from conversations automatically. Its three-tier storage combines a **vector store** (Qdrant by default, with 24+ supported backends including Pinecone, ChromaDB, pgvector, Milvus), an optional **graph store** (Neo4j, Memgraph, or Kuzu for entity-relationship tracking), and **SQLite** for version history. The core pipeline: extract candidate memories via LLM → embed and compare against existing memories → decide ADD/UPDATE/DELETE/NONE. Memories are flat natural-language facts (e.g., "User prefers aisle seats") scoped by user_id, agent_id, and run_id. On the LOCOMO benchmark, Mem0 achieves **26% higher accuracy** than OpenAI Memory with **91% lower p95 latency**. The platform raised **$24M** (October 2025) and claims 80,000+ developers. Graph Memory, available on the Pro tier ($249/month) or via self-hosted Neo4j, adds entity-relationship enrichment to vector search results but does not reorder them—a notable architectural limitation.

**Zep / Graphiti** builds the most sophisticated memory structure: a **temporal knowledge graph** with three tiers. Episode nodes store raw inputs (messages, text, JSON). Semantic entity nodes and fact edges are extracted via LLM-powered NER and relationship extraction. Community nodes aggregate clusters via label propagation. The defining innovation is **bi-temporal modeling**: every edge tracks four timestamps—when a fact entered the system, when it was invalidated, when it was actually true, and when it stopped being true. This enables point-in-time queries ("What was true on March 1?") and non-lossy contradiction handling (old facts are marked invalid, not deleted). Retrieval combines **semantic embeddings, BM25 keyword search, and graph traversal** with a cross-encoder reranker—all without LLM calls at query time, enabling real-time voice agent use cases. Storage is **Neo4j** (primary), with FalkorDB and Kuzu as alternatives. On benchmarks, Zep achieves **94.8% DMR** versus MemGPT's 93.4% and up to **18.5% accuracy improvement** on LongMemEval.

## How the major AI labs handle memory

The three largest AI labs have taken strikingly different approaches to memory, ranging along a transparency spectrum from Anthropic's client-side control to OpenAI's opaque server-side system.

**OpenAI** operates two memory layers in ChatGPT: explicit "saved memories" (a flat list of factual statements like "User is vegetarian") and, since April 2025, implicit **chat history reference** where the entire conversation history can be searched across sessions. Storage backends and retrieval algorithms are not publicly disclosed. The API side is transitioning: the Assistants API (sunset August 2026) stored Threads and Messages server-side, while the replacement Responses API offers stateless calls with optional `previous_response_id` chaining and server-side compaction when context exceeds window limits. No automatic decay exists—memories persist until manually deleted (retained 30 days post-deletion for safety). No branching is supported, though "Temporary Chat" provides isolation.

**Anthropic** took a radically different approach with its **Memory Tool** (beta, September 2025): a client-side filesystem metaphor where Claude creates, reads, and updates files in a `/memories` directory. The developer controls where and how these files are stored—filesystem, database, Redis, anything. This means **no vector search or embedding-based retrieval**; instead, Claude loads the entire memory directory into its 200K-token context window and reasons over it directly. Memory operations appear as visible tool calls, making them inspectable. For Claude Code, memory lives in plain **CLAUDE.md** Markdown files loaded hierarchically (enterprise → project → user). The consumer chatbot gained persistent memory in late 2025, organized into a categorized "Memory summary" (Role & Work, Current Projects, Personal Content). Anthropic reports **39% improvement** combining memory with context editing on agentic search tasks.

**Google's Gemini** offers the most structured consumer memory among the three labs. The `user_context` document organizes memories into typed categories (demographics, interests, relationships, dated events) with each statement carrying a **timestamped rationale** citing the source conversation—e.g., "User explicitly stated on June 18, 2025: '...'" This temporal metadata enables conflict resolution and time-aware reasoning. Context caching provides a separate developer-facing memory mechanism: explicit caches with configurable TTL (90% token cost discount) and implicit caches applied automatically (75% discount, no storage cost). Project Mariner, Google's web browsing agent, maintains only in-session state with no documented persistent cross-session memory.

## The gap between what agents need and what databases offer

The synthesis across all 11 systems reveals six fundamental capabilities that agent memory requires but no current database provides natively.

**Unified multi-modal retrieval is the most acute gap.** Agent memory requires vector similarity, keyword search, graph traversal, temporal filtering, and importance scoring executed in a single query, in one round-trip, within one transaction boundary. Today's production systems stitch together 3–5 databases—Mem0 combines Qdrant + Neo4j + SQLite; teams commonly run Pinecone + Neo4j + Redis + PostgreSQL. As SurrealDB's CEO stated (February 2026): "People are running five different databases and wondering why they can't get good accuracy. It's because they're having to send five different queries." Each additional network hop compounds latency on the agent's critical reasoning path.

**Native temporal modeling doesn't exist in standard databases.** Zep's bi-temporal approach—tracking four timestamps per fact (system creation/expiration, real-world validity/invalidity)—is fundamental to agent memory but implemented entirely as middleware over Neo4j. Traditional knowledge graphs treat facts as static; they can store "Lisa Su is AMD's CEO" but cannot answer "Was she CEO in 2020?" without custom application logic. No production database offers bi-temporal indexing as a first-class primitive.

**Automatic forgetting remains, per multiple practitioners, the hardest unsolved challenge.** Of 11 systems surveyed, only three implement any form of decay: Generative Agents (soft exponential decay via retrieval scoring), MaRS (six formal forgetting policies), and Zep (edge invalidation). No database has built-in decay functions, importance-weighted retention curves, or automatic consolidation. Developers implement `relevance *= 0.99` in Python, run periodic cleanup scripts, or accept unbounded memory growth. The MaRS research demonstrates that a Hybrid forgetting policy (combining temporal, importance, and privacy signals) significantly outperforms naive approaches—but this sophistication exists only in a research paper with no public implementation.

**Memory-state branching is almost entirely absent.** Only LangGraph offers true fork-and-replay capability via its checkpoint system—loading any prior state, modifying it, and creating a new execution branch while preserving the original history. Every other system evolves memory along a single timeline. The UCSD position paper "Multi-Agent Memory from a Computer Architecture Perspective" (March 2026) identifies this as the **"largest conceptual gap"** in multi-agent systems, drawing explicit parallels to computer architecture consistency models (sequential, eventual, causal consistency) that agents need but lack.

**Cross-system ACID transactions are fragile.** Oracle's developer blog (February 2026) highlights that agent memory operations often touch multiple data types simultaneously—updating a vector embedding, modifying a graph relationship, and changing relational metadata must all succeed or all fail. With the Frankenstack approach (multiple databases), partial failures leave agents in inconsistent states. MongoDB's vector search runs on a separate process from document writes, "creating the same consistency risks as a fully fragmented stack."

**Multi-agent memory access has no formal protocol.** When Agent A writes while Agent B reads, what happens? When should an agent's writes become visible to others? Can one agent access another's long-term memory? The field has no answers. Most frameworks use ad-hoc approaches—Mem0 scopes by user_id/agent_id but has no concurrent access controls; LangGraph isolates state per thread with no cross-thread sharing mechanism. The UCSD paper argues agents need formal consistency models analogous to cache coherence protocols in multiprocessor systems.

## Emerging solutions racing to close the gap

Three convergent trends are reshaping the landscape. **Multi-model databases** like SurrealDB 3.0 (GA February 2026, $44M raised) attempt to unify relational, document, graph, vector, and time-series data in one ACID-transactional engine, promising single-query hybrid retrieval. Oracle Database 26ai makes similar claims for enterprise workloads. The PostgreSQL ecosystem is converging around pgvector + TimescaleDB hypertables + standard tables as a pragmatic unified stack. **Memory middleware** platforms (Mem0, Zep/Graphiti, Letta) continue maturing, with Mem0 at 80,000+ developers and Zep's Graphiti at 20K+ GitHub stars. And at the hardware level, **NVIDIA's Inference Context Memory Storage Platform** (announced CES 2026) targets KV cache offloading via BlueField-4 DPUs—Jensen Huang called it "likely the largest storage market in the world, holding the working memory of the world's AIs."

A provocative counterpoint emerged in March 2026: Google's open-sourced "Always-On Memory Agent" uses **just SQLite + Gemini Flash-Lite** with no vector database or embeddings—the LLM reads, thinks, and writes structured memory directly. This works for small-to-medium memory loads but cannot scale to the demands of production multi-agent systems.

## Conclusion

The 11 systems surveyed span a maturity spectrum from research prototypes (Generative Agents, MaRS) through open-source frameworks (Letta, A-MEM, LangGraph) to production platforms (Mem0, Zep) and proprietary lab implementations (OpenAI, Anthropic, Google). Three insights stand out that aren't obvious from examining any single system.

First, **the field has converged on cognitive science categories** (episodic, semantic, procedural memory) as the organizing taxonomy, but implementations diverge completely on how to store and retrieve across these types. CoALA established the vocabulary; everyone adopted the words while building incompatible systems.

Second, **forgetting is the defining unsolved problem**. Only MaRS treats it as a first-class primitive, and MaRS has no public implementation. Production systems either grow memory unboundedly (Letta, A-MEM), implement soft decay via retrieval scoring (Generative Agents), or rely on LLM-driven conflict resolution (Mem0). The gap between human memory's elegant forgetting and AI agent memory's crude accumulation is the field's most important open research question.

Third, **the winning database architecture likely doesn't exist yet**. The middleware approach (Mem0/Zep/Letta sitting above existing databases) adds latency and complexity. The multi-model approach (SurrealDB/Oracle attempting everything in one engine) is unproven at scale. The PostgreSQL-convergence approach (pgvector + extensions) is pragmatic but bolted-together. The system that natively combines vector similarity, graph traversal, bi-temporal indexing, automatic decay, memory branching, and multi-agent consistency within a single transactional engine—at sub-millisecond latency—will likely define the next era of AI infrastructure.
