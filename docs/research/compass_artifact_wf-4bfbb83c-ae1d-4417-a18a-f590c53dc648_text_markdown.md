# Embedded event sourcing with multi-model projections

**No existing system provides automatic event projections into KV, document, graph, and vector models within a single embedded database — but the building blocks are close.** SurrealDB and ArcadeDB offer native multi-model storage (documents, graphs, key-value, vectors) in embeddable engines with ACID transactions, yet neither includes a built-in event sourcing or projection framework. The practical path forward is pairing one of these multi-model engines with a custom projection dispatcher — a single append-only event log fanning out to typed projection handlers, each materializing into a different data model within the same embedded process. This architecture is genuinely novel. The individual components — event stores, projection mechanics, multi-model databases, and event-driven graph construction — are all well-established, but their synthesis into a unified embedded system remains unimplemented.

Event sourcing in embedded/local-first contexts has matured significantly. Libraries exist across Rust, Python, Go, TypeScript, .NET, and Elixir for SQLite-backed event stores. Marten (PostgreSQL) and Axon Framework (JVM) demonstrate production-grade projection systems that operate in single-process mode without message brokers. The key insight from this survey: **CQRS and event sourcing are fundamentally about code organization, not infrastructure** — separate read/write models, in-process event routing, and synchronous projections in the same database transaction eliminate the need for Kafka, RabbitMQ, or any distributed streaming platform.

## Projections are just left-folds over event streams

Every event sourcing projection reduces to a single functional concept: a left-fold. Given an initial state, an ordered sequence of events, and a pure apply function, the projection computes `State = fold(initialState, events, applyEvent)`. This formulation — formalized in Jérémie Chassaing's "Decider" pattern — makes projections composable, testable, and deterministically replayable.

The concrete pipeline in any system follows four stages: events are appended to an immutable log; a subscription mechanism (push or pull) delivers events to projection handlers; each handler applies business logic to compute new read-model state; and the updated state is persisted to a query-optimized store. **Checkpoint tracking** ties this together — each projection records the last event position it has processed, enabling resume-after-failure and catch-up replay. The recommended pattern stores checkpoints in the same transaction as the read-model update, achieving effectively exactly-once semantics without distributed coordination.

Projection lifecycles follow a universal pattern: **create → catch-up → live**. A new projection starts with no checkpoint, replays all historical events in batch mode, then transitions to real-time processing. The "gap problem" — events appended during the transition from batch replay to live subscription — is handled by buffering live events during catch-up and draining the buffer before switching modes. If the projection falls behind again, it automatically re-enters catch-up. Idempotency is achieved through checkpoint-based deduplication (skip events at or before the stored position), naturally idempotent operations (upserts), or embedding the last processed sequence number directly in each read-model row.

## EventStoreDB, Marten, and Axon each solve projections differently

**EventStoreDB** runs server-side projections written in JavaScript, executing on the leader node via a dedicated thread pool. It provides four selectors — `fromAll()`, `fromStream()`, `fromCategory()`, and `fromStreams()` — combined with partitioning operators like `foreachStream()` and `partitionBy()`. Projections output to new event streams via `emit()` (new events) or `linkTo()` (references to existing events). Five built-in system projections handle cross-cutting concerns: `$by_category` links events into category streams, `$by_event_type` groups by event type, and `$stream_by_category` indexes stream existence. Projection modes include **continuous** (runs indefinitely, survives restarts), **oneTime** (processes existing events then stops), and **transient** (in-memory, no persistence). For read-model construction, EventStoreDB relies on client-side catch-up subscriptions where the subscriber manages its own checkpoint — typically storing commit positions and resuming from them on reconnection. Persistent subscriptions offer server-managed checkpoints with competing consumers but sacrifice ordering guarantees.

**Marten** takes a radically different approach by making PostgreSQL serve as both event store and document database. Events are stored as JSONB in an `mt_events` table with a sequential `seq_id` for global ordering. Projections materialize into Marten document tables (also JSONB) or flat relational tables — all within the same PostgreSQL database. The critical differentiator is **three projection lifecycles**: inline projections execute within the same ACID transaction as event capture, providing strong consistency at the cost of write latency; async projections run via a background "Async Daemon" that polls `mt_events` for new entries beyond each projection's checkpoint in `mt_event_progression`; and live projections compute state on demand by replaying events each time, with no persistence. Marten's `SingleStreamProjection` aggregates events from one stream into one document (stream ID maps to document ID), while `MultiStreamProjection` aggregates across streams using configurable identity mappings. The Async Daemon uses PostgreSQL advisory locks for leader election in multi-node deployments and tracks high-water marks to handle gaps from uncommitted transactions.

**Axon Framework** provides a complete CQRS/ES toolkit in the JVM. Projections are Spring components containing `@EventHandler` methods (update read models) and `@QueryHandler` methods (serve queries). The key architectural choice is between **Subscribing Event Processors**, which run in the publishing thread's transaction for synchronous consistency, and **Tracking Event Processors**, which use independent threads with persistent `TrackingToken` positions for async processing with replay support. Tracking tokens are stored via `JpaTokenStore` or `JdbcTokenStore` — **best practice places tokens in the same database as the projection** for transactional atomicity. Replay works by stopping the processor, calling `resetTokens()`, and restarting — the `@ResetHandler` annotation marks methods that clear projection state before replay, and `@DisallowReplay` skips handlers that would cause side effects. Crucially, **Axon works fully without Axon Server**: excluding the `axon-server-connector` dependency auto-configures an `EmbeddedEventStore` backed by JPA or JDBC, with in-process `SimpleCommandBus` and `SimpleEventBus`. Events, tokens, snapshots, and projections can share a single H2 or PostgreSQL database.

## SQLite is a surprisingly capable embedded event store

A growing ecosystem of libraries implements event sourcing on SQLite across multiple languages. The **Python `eventsourcing` library** (v9.5) provides first-class SQLite support with `SQLiteAggregateRecorder`, automatic WAL mode, and a schema using `(originator_id, originator_version)` as primary key with `WITHOUT ROWID` for compact storage. The **Rust `sqlite-es` crate** (v0.4.5) implements `PersistedEventRepository` for the `cqrs-es` framework with events and snapshots tables. In Go, **`plaenen/eventstore`** (v0.0.36) stands out as the most comprehensive: it provides an `EventStore` with idempotent appends, a `CheckpointStore` for projection cursors with transactional save, and a `ProjectionMetadataStore` for schema versions and rebuild flags — plus LibSQL embedded replica support for local-first sync. The TypeScript **`strato-db`** (v3.11.1) is production-proven with multi-GB databases, describing itself as "Server Side Redux" — it layers `EventSourcingDB` and `ESModel` over SQLite with sub-millisecond query times. In .NET, **`EventFlow.SQLite`** provides a full CQRS/ES framework with SQLite persistence for events, read models, and snapshots.

The consensus schema pattern across all implementations uses an append-only events table with `(stream_id, version)` as a unique constraint for optimistic concurrency, `AUTOINCREMENT` on the primary key for global ordering, and JSON payloads. SQLite's single-writer model is actually advantageous for event ordering — unlike PostgreSQL where concurrent transactions can interleave inserts causing sequence gaps, **SQLite serializes all writes, guaranteeing insert order equals commit order**. WAL mode is essential, enabling unlimited concurrent readers (projections) alongside the single writer. Recommended pragmas for event sourcing workloads: `journal_mode=WAL`, `busy_timeout=5000`, `synchronous=NORMAL`, and sufficient `cache_size`. The main limitation is lack of built-in pub/sub — projections must poll for new events or use `sqlite3_update_hook()` for per-connection notifications.

## Deterministic replay requires discipline, not magic

Replaying events to reconstruct state is deterministic when — and only when — the apply function is pure. **Four categories of impurity break replay:** side effects (sending emails, API calls), time-dependent logic (`DateTime.Now` in handlers), external state reads (database lookups, cache hits), and non-deterministic operations (random numbers, new GUIDs). The solution is architectural: all information needed for deterministic replay must be captured in the events themselves at command-processing time.

Systems handle this through several established patterns. The **Gateway pattern** wraps external systems and checks whether the processor is in replay mode before forwarding calls. The **Noop Component pattern** replaces side-effect-triggering components with no-op implementations during replay. Axon's `@DisallowReplay` annotation skips specific handlers entirely during replay, while `ReplayToken` wrapping lets handlers branch on `ReplayStatus`. The deeper principle is the **command/event separation**: commands represent intentions that may trigger side effects; events represent immutable facts. Side effects belong exclusively in command handling, never in event handling. When an event handler needs external data (exchange rates, timestamps, computed values), that data should have been fetched during command processing and embedded in the event payload.

**Schema evolution** is the other replay challenge. Events are immutable but schemas change over time. The most widely adopted technique is **upcasting** — middleware that transforms old event formats into the current format at read time, before events reach application logic. Upcasters chain (v1→v2→v3), so only the latest handler version needs maintenance. Alternatives include weak schema / tolerant reader patterns (ignore unknown fields, default missing ones), versioned event types with per-version handlers, and copy-and-transform migrations for major overhauls. **Snapshots** optimize replay by caching aggregate state at a position — subsequent loads replay only events after the snapshot. A typical threshold is snapshotting every 100–1000 events, with the snapshot versioned to handle invalidation when aggregate logic changes.

## In-process CQRS eliminates distributed complexity entirely

CQRS without a message broker is not only possible — Greg Young, who coined the term, emphasized that CQRS is a five-minute concept requiring no distributed infrastructure. **Single-process, single-database CQRS** maintains separate code models for reads and writes sharing one database, with both updated in the same transaction for strong consistency. The command bus becomes a direct method call. The event bus becomes an in-process observer pattern. The query bus becomes simple function dispatch.

The single-writer model dramatically simplifies the architecture. There are no distributed transactions, no message ordering concerns, no exactly-once delivery problems, no token claiming contention, and no eventual consistency to reason about. Marten's inline projections exemplify this: events are appended and projections updated within a single PostgreSQL ACID transaction. Axon's `SubscribingEventProcessor` achieves the same — event handlers run in the publishing thread's transaction, and if projection logic fails, the event append rolls back too.

The pragmatic guidance from the community is to **start with the simplest viable CQRS** — single process, single database, synchronous projections — and evolve toward distribution only when concrete requirements demand it. This means you can get the full architectural benefits of event sourcing (complete audit trail, temporal queries, projection rebuilds, event-driven read models) with nothing more than a SQLite file and application code. The "graduation" to distributed infrastructure happens when write throughput exceeds single-process capacity, when different read models need different storage technologies, or when teams require independent deployability.

## No single system handles multi-model projection today, but the architecture is clear

After thorough investigation, **no existing system provides automatic event projections materializing into KV + document + graph + vector models within a single embedded database.** This represents a genuine gap. The closest approximations come from two directions: multi-model databases that lack event sourcing frameworks, and event sourcing frameworks that target single-model outputs.

**SurrealDB** is the strongest candidate for the underlying engine. It natively supports documents, graphs (via `RELATE`), key-value, vectors (HNSW indexing), and time-series — all within ACID transactions, all embeddable in Rust, Python, JavaScript, and WebAssembly. Its `DEFINE EVENT` triggers and live queries provide reactive primitives for projection dispatch. **ArcadeDB** (JVM, Apache 2.0) offers comparable coverage: graph, document, key-value, vector, and time-series with **2M+ inserts/sec** in embedded mode. **SQLite with extensions** (sqlite-vec for vectors, adjacency tables for graphs, JSON1 for documents) provides maximum portability but non-native graph traversal.

The theoretical architecture is straightforward:

A single append-only event log table serves as the source of truth. A projection dispatcher reads events sequentially (or reacts to triggers) and fans out to typed handlers. Each handler — KV, Document, Graph, Vector — maintains its own checkpoint and materializes its model within the same database. In SurrealDB, this means a `UserCreated` event simultaneously executes `INSERT INTO kv_store`, `INSERT INTO user_documents`, `RELATE user:123->member_of->org:456`, and `INSERT INTO embeddings` with HNSW-indexed vectors — all in one ACID transaction if using synchronous projection, or with per-handler checkpoints if async.

The vector projection poses a unique challenge: generating embeddings typically requires an ML model, introducing non-determinism. The solution is to **compute embeddings at ingestion time and store them in the event payload**, making the vector projection handler a simple write of pre-computed data rather than an inference step. This preserves deterministic replay.

For event-driven knowledge graph construction specifically, the pattern maps naturally: events become edges (temporal, causal connections), entities mentioned in events become nodes, and event metadata becomes edge properties. **Telicent CORE** is the most mature implementation of this pattern, using Kafka → Apache Jena with RDF-formatted events. The **FinKario** system (2025) demonstrated LLM-driven entity extraction from financial events into a dual-schema knowledge graph with 305K+ entities. In an embedded context, the graph projection handler would use MERGE/upsert operations for idempotency and map event types to graph operations — `UserCreated` creates a node, `PurchaseMade` creates an edge between user and product nodes.

## Conclusion

The embedded event sourcing ecosystem is more mature than its distributed counterpart might suggest. Production-quality libraries exist for SQLite (Python, Rust, Go, TypeScript, .NET), Marten provides a complete event store plus projection engine requiring only PostgreSQL, and Axon Framework runs fully embedded in a single JVM with no external infrastructure. The core mechanics — append-only logs, checkpoint-tracked projections, catch-up subscriptions, deterministic replay — are well-understood and consistently implemented across all these systems.

The genuinely novel frontier is **multi-model projection within a single embedded process**. SurrealDB's combination of document, graph, key-value, and vector models in an embeddable ACID engine makes it the most promising substrate — what's missing is the projection framework layer: event log management, checkpoint tracking, handler registration, and replay orchestration. This is an engineering project, not a research problem. The FoundationDB "layers" pattern (demonstrated at scale by SkuVault with 1.5B events across multiple data models in unified transactions) proves the concept works; the contribution would be bringing it to an embedded, local-first context where the entire system — event log, four heterogeneous read models, projection engine — runs in a single process backed by a single database file.
